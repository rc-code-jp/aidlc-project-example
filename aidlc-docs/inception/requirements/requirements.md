# Requirements

## Intent Analysis Summary
- **User Request**: AI-DLC を使って、MVP 実装の犬判定 Web アプリの UI を改善し、よりユーザー体験の良い Web アプリにする
- **Request Type**: Enhancement
- **Scope Estimate**: Single Component
- **Complexity Estimate**: Moderate

## Solution Summary
- 現在のトップページ UI を対象に、見た目の印象を大きく引き上げる
- アップロード体験に画像プレビューとドラッグアンドドロップを追加する
- 判定結果をより視覚的に理解しやすい表現へ改善する
- 既存の API、Gemini 連携、業務ルールは維持する
- 今回の変更範囲はトップページ中心に限定し、全面的な構造変更は行わない

## Functional Requirements

### FR-1 アップロード導線の改善
- ユーザーはトップページ上で画像を 1 枚アップロードできること
- 画像選択は従来のファイル選択に加えて、ドラッグアンドドロップでも行えること
- 画像を選択した後、送信前に選択画像のプレビューを確認できること
- 画像未選択時は判定操作を開始しないこと

### FR-2 判定処理の継続
- アプリは既存どおりアップロード画像をサーバー側 API 経由で Gemini API に送信し、犬らしさを 0 から 100 のパーセントで取得できること
- サーバー側は既存の業務ルールを維持すること
  - 本物の犬が写っている場合は 100%
  - 犬っぽい物体、表現物、類似対象が写っている場合は 100 未満の適切な値
  - 犬と無関係な画像は低い値
- Gemini API の応答は、少なくとも `percentage` と `label` を含む構造化結果として扱うこと

### FR-3 結果表示の視覚強化
- 画面は犬らしさのパーセントを強調表示すること
- 画面は判定ラベルとして `犬`、`犬っぽい`、`犬ではない` のいずれかを表示すること
- 結果は単なるテキスト列ではなく、視覚的に理解しやすいレイアウトや表現で示すこと
- ユーザーが別画像をアップロードした際は、新しい結果で表示を更新すること

### FR-4 状態表示の改善
- 判定中、成功、失敗、待機中の各状態が現在よりも認識しやすく表示されること
- エラー時は再試行しやすい状態を維持すること

### FR-5 既存の安全性維持
- アプリは判定に使用した画像ファイルを永続保存しないこと
- 判定完了後はサーバー側に画像を保持し続けないこと
- Gemini API キーはブラウザへ露出させず、サーバー側のみで利用すること

## Non-Functional Requirements

### NFR-1 UX / Visual Quality
- MVP 感の強い単一カード UI から脱却し、第一印象が明確に向上すること
- デスクトップとモバイルの両方で破綻なく表示できること
- 判定前、判定中、判定後で視線誘導が分かりやすいこと

### NFR-2 Usability
- 画像選択から結果確認までの操作が直感的であること
- ドラッグアンドドロップやプレビューがあっても、操作手順が複雑になりすぎないこと

### NFR-3 Maintainability
- 改善対象は主にトップページとスタイルに集中させること
- UI 改善に伴い必要な範囲でテストを補強できること

### NFR-4 Security
- Gemini API キーは環境変数などの安全な設定で管理すること
- ブラウザから直接外部 AI API を呼ばないこと

## User Scenarios
- ユーザーが画像をドラッグアンドドロップすると、判定前にプレビューが表示される
- ユーザーが犬の写真をアップロードすると、視覚的に強調された `100%` と `犬` の結果が表示される
- ユーザーが犬のぬいぐるみや犬のイラストをアップロードすると、視覚的に分かりやすい結果 UI で `100% 未満` の値と `犬っぽい` が表示される
- ユーザーが犬ではない風景や物体をアップロードすると、低い値と `犬ではない` が分かりやすく表示される
- 外部 API 呼び出しに失敗した場合、画面に認識しやすいエラーメッセージが表示され、再試行しやすい状態が維持される

## Constraints and Decisions
- 今回の主対象はトップページ UI の改善である
- API エンドポイント、Gemini 呼び出し方式、判定ロジックの本質は維持する
- 画像履歴保存、ユーザー認証、複数ページ化は今回の対象外とする

## Open Items
- 判定結果の具体的な視覚表現は Workflow Planning と実装設計で確定する
- 必要なコンポーネント分割と UI テスト拡張の範囲は後続フェーズで最終化する
