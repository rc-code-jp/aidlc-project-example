# Requirements

## Intent Analysis Summary
- **User Request**: 画像をアップロードすると、その画像に犬が写っているかを判定し、犬らしさをパーセント表示する Web アプリを作る。犬が写っていれば 100%、犬っぽい物が写っている場合はパーセントを増やす。
- **Request Type**: New Project
- **Scope Estimate**: Single Component から Multiple Components 相当
- **Complexity Estimate**: Moderate

## Solution Summary
- 単一ページのシンプルな Web アプリとして提供する
- ユーザーがアップロードした画像をサーバー側 API 経由で Gemini API に送信する
- Gemini にはシステムプロンプトで判定仕様を与え、犬らしさのパーセントと判定ラベルを返させる
- 画像は判定後に保存せず破棄する
- UI にはパーセントに加えて、`犬`、`犬っぽい`、`犬ではない` の文言を表示する
- 初期リリースはローカルで動く最小構成を対象とする

## Functional Requirements

### FR-1 画像アップロード
- ユーザーはブラウザ画面から 1 枚の画像をアップロードできること
- アプリは画像選択後に判定を実行できること

### FR-2 犬判定
- アプリはアップロード画像を Gemini API へ送信し、犬らしさを 0 から 100 のパーセントで取得できること
- サーバー側は Gemini API に対し、以下の業務ルールを含むシステムプロンプトを送ること
  - 本物の犬が写っている場合は 100%
  - 犬っぽい物体、表現物、類似対象が写っている場合は 100 未満の適切な値
  - 犬と無関係な画像は低い値
- Gemini API の応答は、少なくとも `percentage` と `label` を含む構造化結果として扱うこと

### FR-3 判定結果表示
- 画面は犬らしさのパーセントを表示すること
- 画面は判定ラベルとして `犬`、`犬っぽい`、`犬ではない` のいずれかを表示すること
- ユーザーが別画像をアップロードした際は、新しい結果で表示を更新すること

### FR-4 画像データの非保持
- アプリは判定に使用した画像ファイルを永続保存しないこと
- 判定完了後はサーバー側に画像を保持し続けないこと

### FR-5 API キー保護
- Gemini API キーはブラウザへ露出させず、サーバー側のみで利用すること

## Non-Functional Requirements

### NFR-1 セキュリティ
- Gemini API キーは環境変数などの安全な設定で管理すること
- ブラウザから直接外部 AI API を呼ばないこと

### NFR-2 使いやすさ
- 画像選択から結果表示までの流れは単純で、単一画面で完結すること
- 判定中、成功、失敗の状態がユーザーに分かること

### NFR-3 保守性
- フロントエンドと Gemini 呼び出し処理の責務を分離しやすい構成にすること
- 犬判定プロンプトはコード上で差し替え・調整しやすい形で管理すること

### NFR-4 ローカル実行性
- 初期版はローカル環境で起動し、動作確認できること
- 将来的にクラウド展開へ拡張しやすい構成であることが望ましい

## User Scenarios
- ユーザーが犬の写真をアップロードすると、`100%` と `犬` が表示される
- ユーザーが犬のぬいぐるみや犬のイラストをアップロードすると、`100% 未満` の値と `犬っぽい` が表示される
- ユーザーが犬ではない風景や物体をアップロードすると、低い値と `犬ではない` が表示される
- 外部 API 呼び出しに失敗した場合、画面に再試行可能なエラーメッセージを表示する

## Constraints and Decisions
- 画像判定エンジンは Gemini API を利用する
- Gemini API 呼び出しはサーバー側 API 経由で実装する
- 初期段階では画像履歴保存やユーザー認証は対象外とする

## Open Items
- Gemini API に返させる JSON スキーマの厳密形式は実装設計時に確定する
- フロントエンドとバックエンドの具体的な技術スタックは Workflow Planning 以降で最終化する
